Built for a threat most blockchains ignore

Bitcoin, Ethereum, and nearly every other chain sign transactions using elliptic-curve cryptography — secure against ordinary computers, but not against a sufficiently powerful quantum computer. That's not a hypothetical: adversaries can record public blockchain data today and decrypt it once the hardware catches up, a strategy known as "harvest now, decrypt later."

Quantum-Lattice signs every transaction with ML-DSA-65 — the finalized NIST post-quantum standard (FIPS 204) — built specifically to resist both classical and quantum attacks.

Choose how to mine

Solo mining pays the full reward, rarely. Pool mining pays smaller, steadier amounts, far more often. Both use the exact same program — just a different command, shown below.

Solo

Mine directly against the network

Keep 100% of any block you find — but at current network difficulty, that can mean a long wait between wins.

./miner quantum-lattice.futuristicai.co.za:8034 YOUR_WALLET_ADDRESS
Pool

Contribute to the shared pool

Get paid a proportional share every time the pool finds a block — smaller amounts, but far more frequent and predictable.

./miner --pool ql-pool.futuristicai.co.za:7999 YOUR_WALLET_ADDRESS

On Windows, replace ./miner with .\miner.exe. Both modes automatically use every CPU core available on your machine.

Security & Transparency

What's actually been reviewed, and real, independently reproducible cryptographic test vectors — not just claims.

What's been reviewed

The codebase has gone through careful, methodical internal review — consensus rules and transaction validation, cryptographic signing and verification, vault encryption at rest, network input handling on every public endpoint, and abuse protection (rate limiting) on anything that sends email or accepts public input. Issues found during that process were fixed and re-tested before release.

Consensus & Ledger

Every transaction's signature is verified before funds move. Block validation, difficulty retargeting, and supply issuance all run identically whether a block was mined locally, received from a peer, or pulled during catch-up sync.

Wallet & Vault Security

Wallet keys are generated and signed entirely client-side — the node never sees a private key. Treasury vault keys are encrypted at rest (AES-256-GCM + PBKDF2) and unlocked once into memory at startup, never re-read from disk in plaintext.

Network & Abuse Protection

Every public-facing endpoint validates its input before acting on it. Anything that sends email or accepts unauthenticated submissions is rate-limited by both IP and account, to prevent abuse of the service or of anyone's inbox.

Cryptographic test vectors

Real, reproducible output — not a claim. Each of these uses a fixed, published input; run the same computation yourself and you'll get the exact same result. This is standard practice for verifying a cryptographic implementation independently, the same way NIST itself publishes known-answer tests for ML-DSA.
